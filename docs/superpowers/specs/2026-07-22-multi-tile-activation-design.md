# 멀티 타일 동시 작동 설계

작성일: 2026-07-22
대상: 서브 프로젝트 ① — 낙상 방향에 따라 복수의 충격 완화 타일을 동시에 작동시킨다

---

## 1. 배경

낙상 직전을 예측해 바닥의 충격 완화 타일을 작동시키는 프로젝트다. 현재 `main.py`는
MediaPipe로 포즈를 추정하고, 학습된 분류기(`fall_risk_model.joblib`)로 낙상 위험을
판정한 뒤, 호모그래피로 발 위치를 타일 격자에 매핑해 아두이노에 신호를 보낸다.

현재 구조의 한계는 세 가지다.

**타일이 한 번에 하나만 작동한다.** `pixelToTile()`이 발 위치에 해당하는 타일 하나만
반환하고(`main.py:117`), 시리얼은 문자 하나(`'1'`~`'4'`)만 보낸다(`main.py:168`).
실제 낙상에서 몸은 여러 타일에 걸쳐 착지하므로 한 장으로는 충격 완화가 성립하지 않는다.

**낙상 방향 정보가 코드에 존재하지 않는다.** `tiltAngleDeg()`가 `abs(dx)`, `abs(dy)`로
부호를 버려서(`main.py:82`) 기울기의 크기만 남고 방향이 사라진다. MediaPipe가 주는
깊이값 `z`도 수집만 하고(`main.py:59`) 한 번도 사용하지 않는다.

**프로토콜에 확인 수단이 없다.** `arduino.write()`가 던지고 끝이라(`main.py:174`) 신호가
전달됐는지 알 방법이 없다. 시리얼 연결 시 아두이노가 리셋되는데 핸드셰이크가 없어
부팅 중에 보낸 신호는 조용히 사라진다.

## 2. 목표와 범위

### 목표

1. 낙상 방향에 따라 타일 **1장 / 2장 / 3장 / 4장**을 동시에 작동시킨다.
2. 방향 판정이 불가능한 상황(수직 붕괴, 판정 불안정)에서는 안전하게 전체를 작동시킨다.
3. 시리얼 프로토콜에 응답과 핸드셰이크를 도입해 상태를 확인 가능하게 만든다.
4. 시연 당일 실패에 대한 다단계 대비를 갖춘다.

### 범위 밖

- **웹 시뮬레이터** — 설계는 마쳤으나 우선순위 조정으로 보류. 별도 문서로 남긴다.
- **앱(사용자용/보호자용)** — 서브 프로젝트 ②.
- **동선 히트맵** — 서브 프로젝트 ④.
- **낙상 예측 모델 자체의 개선** — 기존 `fall_risk_model.joblib`을 그대로 사용한다.
  모델의 입력 특징(`vy, vx, tilt, tilt_vel`)은 **변경하지 않는다**. 학습/서빙 불일치가
  생기기 때문이다. 이번 작업의 방향 추정값은 모델 입력이 아니라 별도 계산값이다.

## 4. 하드웨어 구성

타일 4장, 2×2 격자. 번호는 기존 `calibrate.py`의 규칙(행 우선)을 따른다.

```
0  1        row 0 = 카메라에서 먼 쪽
2  3        row 1 = 카메라에 가까운 쪽
```

서보는 PCA9685 채널 `{1, 3, 5, 7}`에 연결되어 있고, 외부 5V 어댑터로 구동한다.
V+와 GND 사이에 1000µF 전해 커패시터를 추가한다(기동 전류에 의한 전압 강하 완화).

**타일 번호와 PCA9685 채널의 매핑은 펌웨어 안에만 존재한다.** 파이썬, `calibration.json`,
로그, 시리얼 프로토콜은 전부 타일 번호 0~3만 사용한다.

## 5. 모듈 구조

현재 `main.py`는 337줄의 최상위 스크립트로, 함수로 감싸지 않은 전역 루프 안에서
영상 입출력·포즈 추정·특징 계산·모델 추론·타일 기하·시리얼 통신·로깅·렌더링이
모두 섞여 있다. 전면 재작성은 하지 않되 이번 작업에 걸리는 부분만 분리한다.

```
tiles.py           방향 추정 + 타일 선택       의존성: math
tile_protocol.py   시리얼 프로토콜             의존성: pyserial
pose_source.py     카메라 입력 어댑터          의존성: mediapipe, opencv, joblib
calibration.py     호모그래피 로드/렌더링      (기존 main.py에서 이동)
main.py            조립부 (얇아짐)
hardware/four_servo_control/four_servo_control.ino   펌웨어
```

**설계상 핵심 성질: `tiles.py`와 `tile_protocol.py`는 mediapipe, opencv, 카메라,
인형 어느 것에도 의존하지 않는다.**

이 성질이 세 가지를 보장한다.

- 두 모듈이 전체 로직의 핵심을 담으면서도 **하드웨어 없이 단위 테스트로 완전히 검증**된다.
- 가정 A1이 미검증인 채로도 구현의 대부분(§12의 1~4단계)이 진행된다.
- 나중에 입력 어댑터(웹 시뮬레이터, 앱)를 추가할 때 로직을 복제할 자리가 없다.
  `tiles.select_tiles()`를 호출하는 것 외의 경로가 존재하지 않는다.

## 6. 방향 추정과 타일 선택

### 6.1 방향 계산

**어깨 중점 → 엉덩이 중점 벡터**를 사용한다. 발목(랜드마크 27, 28)은 사용하지 않는다.
집 모형 내부에서 가려지기 쉽고 인형에서 인식되지 않을 위험이 크며, 어깨·엉덩이는
낙상 모델의 입력이라 어차피 잘 인식되어야만 하는 값이기 때문이다.

```
dx = 어깨중점.x − 엉덩이중점.x     이미지 픽셀,  + = 오른쪽
dy = 어깨중점.y − 엉덩이중점.y     이미지 픽셀,  + = 아래
dz = 어깨중점.z − 엉덩이중점.z     MediaPipe 깊이, + = 카메라에서 멀어짐
```

`dz`가 앞/뒤 낙상을 판별한다. 이 값은 `main.py:59`에서 이미 수집하고 있으나 현재
사용되지 않는다.

바닥 평면 기준 방향을 다음과 같이 정의한다.
**0° = 격자 위쪽(카메라에서 먼 쪽), 시계방향 증가.** 따라서 90°=우, 180°=앞, 270°=좌.

```
direction_deg = (degrees(atan2(dx, dz)) − camera_yaw_deg) mod 360
lean_ratio    = sqrt(dx² + dz²) / sqrt(dx² + dy² + dz²)
```

`lean_ratio`는 기울기의 사인값이다. 0 = 직립, 1 = 완전히 누움. 화면상 크기에 무관하므로
인형과 사람에 같은 임계값을 적용할 수 있다.

`camera_yaw_deg`는 카메라 광축과 타일 격자 축의 각도 차이로, `calibration.json`에
추가한다. 기본값 0.

**기존 `tiltAngleDeg()`(`main.py:82`)는 수정하지 않는다.** 학습된 모델의 입력 특징이므로
정의를 바꾸면 학습/서빙 불일치가 발생한다(`main.py:73`의 기존 주석도 같은 경고를 한다).
`lean_ratio`는 그 옆에 새로 추가되는 별개 값이다.

### 6.2 판정은 발사 시점에 한 번만

프레임마다 방향을 재판정하면 구간 경계에서 값이 진동한다. 최근 N프레임(§8의 프로파일
`window`, 기본 8)의 `(direction_deg, lean_ratio)`를 큐에 유지하고,
`consecutive_risk_frames >= persistence`로 발사가 확정되는 순간(`main.py:297`)에
한 번만 계산한다.

큐에는 **랜드마크가 유효한 프레임만** 넣는다. 추적을 놓치면 기존 동작과 동일하게
큐를 비운다(`main.py:308`~`316`이 `vy_hist` 등에 대해 하는 것과 같다). 오래된 방향이
새 낙상의 판정에 섞이면 안 된다.

각도이므로 산술평균은 틀린다(350°와 10°의 평균이 180°가 된다). 원형 평균을 사용한다.

```
mean_direction = atan2( Σ sin θᵢ , Σ cos θᵢ )
R              = |Σ (cos θᵢ, sin θᵢ)| / N        → 0~1, 방향 일치도
```

`R`은 원형통계의 평균 결과 길이다. N개 표본이 같은 방향을 가리키면 1에 가깝고,
흩어지면 0에 가깝다. 이 값을 확신도로 사용한다.

### 6.3 타일 선택 — 규칙 세 개

하드코딩 테이블 대신 규칙 세 개로 정의한다. 격자 크기가 바뀌어도 규칙은 유지된다.

- **규칙 1 (정방향, 각 중심 ±22.5°):** 그 방향의 행 또는 열 전체. → 2×2에서 2장
- **규칙 2 (대각, 각 중심 ±22.5°):** 전체 타일에서 반대편 모서리 한 장을 제외. → 3장
- **규칙 3 (대각 정밀, 각 중심 ±10° 이내이고 동시에 `R ≥ τ_R_strict`):** 그 모서리 한 장만. → 1장

**규칙 3이 대각에만 적용되는 것은 2×2의 기하 때문이다.** 정방향 낙상은 1장으로 좁힐
수 없다 — "우측"이라는 정보만으로는 앞뒤 성분을 알 수 없으므로 열 전체를 덮어야 한다.
반면 대각 낙상은 모서리 한 장이 특정되므로, 방향 확신이 충분히 높으면 그 한 장에
집중할 수 있다.

규칙 3에는 조건이 두 개 붙는다. `R ≥ τ_R_strict`는 각도 자체에 대한 확신을 요구하고,
`±10°` 조건은 구간 경계 부근(예: 30°)에서 확신이 높다는 이유만으로 모서리 한 장에
투신하는 것을 막는다.

2×2에 적용한 결과:

| 방향 구간 | 중심 | 조건 | 켜지는 타일 | 장수 |
|---|---|---|---|---|
| 337.5–22.5° | 0° 뒤(먼 쪽) | | `0 1` | 2 |
| 22.5–67.5° | 45° 뒤·우 | 35–55° 그리고 `R ≥ τ_R_strict` | `1` | **1** |
| 22.5–67.5° | 45° 뒤·우 | 그 외 | `0 1 3` (2 제외) | 3 |
| 67.5–112.5° | 90° 우 | | `1 3` | 2 |
| 112.5–157.5° | 135° 앞·우 | 125–145° 그리고 `R ≥ τ_R_strict` | `3` | **1** |
| 112.5–157.5° | 135° 앞·우 | 그 외 | `1 2 3` (0 제외) | 3 |
| 157.5–202.5° | 180° 앞(가까운 쪽) | | `2 3` | 2 |
| 202.5–247.5° | 225° 앞·좌 | 215–235° 그리고 `R ≥ τ_R_strict` | `2` | **1** |
| 202.5–247.5° | 225° 앞·좌 | 그 외 | `0 2 3` (1 제외) | 3 |
| 247.5–292.5° | 270° 좌 | | `0 2` | 2 |
| 292.5–337.5° | 315° 뒤·좌 | 305–325° 그리고 `R ≥ τ_R_strict` | `0` | **1** |
| 292.5–337.5° | 315° 뒤·좌 | 그 외 | `0 1 2` (3 제외) | 3 |
| — | — | 게이트 미통과(§6.4) | `0 1 2 3` | 4 |

**§6.4의 게이트가 규칙 3보다 우선한다.** `R < τ_R` 또는 `lean_ratio < τ_lean`이면
방향과 무관하게 4장이다.

### 6.3.1 1장 작동의 물리적 타당성

1장만 작동시키면 몸의 나머지 부위는 맨바닥에 닿는다. 이것이 충격 완화로서 유효한지는
**타일 실물 크기 대비 인체 크기**에 달려 있다. 타일이 충분히 커서 엉덩이·어깨 같은
주요 충격 지점을 덮는다면 타당하고, 작다면 보호받는다는 착각만 준다.

타일 실측 치수가 확정되면 이 판단을 재검토한다. 규칙 3만 비활성화하면 되며
(`τ_R_strict = 1.0` 이상으로 설정), 나머지 설계는 영향받지 않는다.

### 6.4 게이트 — 4장 승격 조건

방향이 기본 장수(대각 3장 / 정방향 2장)를 결정하고, 확신도가 양쪽 끝을 담당한다 —
**아래로는 §6.3 규칙 3이 대각을 1장으로 좁히고, 위로는 아래 게이트가 전체를 4장으로
넓힌다.** 둘 중 하나라도 걸리면 전체 4장을 작동시킨다.

| 게이트 | 조건 | 잡아내는 상황 |
|---|---|---|
| 방향 일치도 | `R < τ_R` | 프레임마다 방향이 튐 — 판정 불가 |
| 기울기 | `lean_ratio < τ_lean` | 수직으로 주저앉음 — 방향이 존재하지 않음 |

두 게이트는 서로 다른 실패 양상을 잡으므로 둘 다 필요하다. 두 경우 모두 **안전한
방향으로 실패한다** — 판정이 불확실하면 전체를 작동시킨다.

4장 작동은 실패 시의 도피 경로가 아니라 설계된 최상위 등급이다. 수직 붕괴는 착지
지점을 특정할 수 없으므로 전면 작동이 물리적으로 옳다.

### 6.5 임계값

시작값: `τ_R = 0.85`, `τ_R_strict = 0.95`, `τ_lean = 0.15`.

**이 세 값에는 물리적 근거가 없다.** 1장·2장·3장·4장이 고르게 나오도록 조정하는
연출값이며, §10.3의 절차로 리허설 데이터에서 확정한다. 코드 상수가 아니라 프로파일
설정으로 둔다(§8).

`τ_R_strict`가 추가되면서 튜닝해야 할 손잡이가 둘에서 셋으로 늘었다. 다만
`τ_R_strict`는 `τ_R`과 **같은 양(방향 일치도)에 대한 두 번째 기준선**이므로 새로운
측정량을 도입하는 것보다는 부담이 적고, 분포를 한 번 그리면 두 값을 함께 정할 수 있다.
`τ_R ≤ τ_R_strict` 관계는 항상 성립해야 하며, 설정 로드 시 검증한다.

주의할 점: 이 시스템은 낙상 **직전**에 발사하므로 발사 시점의 `lean_ratio`가 체계적으로
작다. 20° 기울었을 때 발사되면 `lean_ratio ≈ 0.34`이다. 시작값을 잘못 두면 모든 낙상이
4장으로 떨어져 방향 타겟팅이 한 번도 관찰되지 않을 수 있다.

### 6.6 알려진 위험 — MediaPipe `z`의 노이즈

MediaPipe의 `z`는 `x`, `y`보다 노이즈가 크다. 앞/뒤 판정이 좌/우보다 불안정할 것으로
예상한다.

설계가 이를 자동으로 흡수한다. `z`가 튀면 프레임 간 방향이 흩어지고 → `R`이 떨어지고
→ 게이트에 걸려 4장이 된다. 잘못된 방향으로 2장을 켜는 것이 아니라 안전하게 전체를 켠다.

**차선책:** 실측에서 앞/뒤 판정이 실용적이지 않으면 `dz`를 버리고 좌/우만 판정하며,
앞뒤 성분은 전부 4장으로 처리한다. 이 경우에도 "좌·우 2장 / 수직 4장" 두 가지는
확보된다. §7 이후를 건드리지 않는 국소 변경이므로 §10.3의 결과를 보고 결정한다.

## 7. 시리얼 프로토콜

줄 단위 ASCII, 종결자 `\n`, 보율 115200.

### 7.1 문법

**파이썬 → 아두이노**

| 명령 | 의미 |
|---|---|
| `FIRE 1,3\n` | 타일 1, 3을 작동 위치(`MOVE_ANGLE`)로 |
| `RESET\n` | 작동 중인 서보 전부 `HOME_ANGLE`로 |
| `PING\n` | 생존 확인 |

**아두이노 → 파이썬**

| 응답 | 의미 |
|---|---|
| `READY 4\n` | 부팅 완료, 사용 가능한 서보 개수 |
| `OK FIRE 1,3\n` | 수행 완료. **받은 인자를 그대로 되돌려준다** |
| `OK RESET\n` | |
| `OK PING\n` | |
| `ERR <사유>\n` | 파싱 실패, 범위 밖 타일 번호 등 |
| `# <텍스트>\n` | 사람이 읽을 주석. 파이썬은 무시 |

`OK`가 인자를 되돌려주므로 파이썬이 보낸 것과 아두이노가 이해한 것의 일치를 확인할 수
있다. 이는 서브 프로젝트 ②의 앱이 "실제 하드웨어 상태"를 표시하기 위한 전제다.

`#` 줄을 무시하는 규칙 덕분에 펌웨어에 사람용 안내문을 남겨도 파서가 깨지지 않는다.
현재 `.ino`가 부팅 시 출력하는 한국어 안내 두 줄은 `#` 접두사를 붙여 유지한다.

### 7.2 연결 핸드셰이크

```
포트 열기 → 입력 버퍼 비우기 → READY 줄 대기 (타임아웃 5초)
  → 서보 개수와 rows×cols 대조 → 불일치 시 경고
  → 준비 완료
```

대부분의 아두이노는 시리얼 포트가 열릴 때 리셋되고 `setup()`에는 `delay(500)`이 있다.
현재 `connect_arduino()`는 포트를 열자마자 반환하므로(`main.py:160`) 부팅 중에 보낸
신호는 사라진다. 핸드셰이크가 이 버그를 제거한다.

`READY n`의 개수를 `calibration.json`의 `rows × cols`와 대조하여, 캘리브레이션과
펌웨어 구성이 어긋난 상태로 시연장에 가는 사고를 방지한다.

### 7.3 타임아웃과 블로킹

응답 대기는 영상 루프를 블로킹한다. 두 가지로 완화한다.

- **보율 9600 → 115200.** `OK FIRE 1,3\n` 13바이트가 9600에서 약 13ms, 115200에서
  약 1ms. `.ino`의 `Serial.begin()`도 함께 변경한다.
- **응답 타임아웃 200ms.** 현재 `timeout=1`은 아두이노 무응답 시 1초간 영상이 멈춘다.
  발사는 `RISK_COOLDOWN = 3.0`초에 한 번뿐이므로 최악 200ms 지연은 허용 가능하다.

**ACK 실패가 시연을 중단시켜서는 안 된다.** 타임아웃 시 경고를 로그에 남기고 계속
진행한다. `fire()`는 예외를 던지지 않고 `bool`을 반환한다.

### 7.4 리셋

현재 동작을 유지한다 — 발사 후 `RESET_DELAY = 2.0`초 뒤 전체 원위치(`main.py:318`).
타일이 여러 장이어도 `RESET` 하나로 전부 복귀한다. 타일별 선택 리셋은 만들지 않는다.

### 7.5 동시성

`FIRE 1,3` 수신 시 펌웨어가 해당 채널에 `pwm.setPWM()`을 순차 호출한다. I2C 트랜잭션
하나가 1ms 미만이므로 4개를 모두 처리해도 수 ms 안에 끝나며 시각적으로 동시다.
외부 5V 전원을 사용하므로 시차 발사(staggering)는 도입하지 않는다(가정 A2).

### 7.6 인덱싱 통일

현재 `main.py:168`의 `motor_id = tile_index + 1` 오프셋은 단일 문자 프로토콜이 `'0'`을
리셋으로 사용해서 생긴 것이다. `RESET`이 별도 단어가 되면서 이 충돌이 사라진다.

**전선 위에 타일 번호 0~3이 그대로 흐른다.** 파이썬, `calibration.json`, 로그,
프로토콜이 모두 같은 번호를 사용한다.

**기존 단일 문자 처리(`'1'`~`'4'`, `'0'`)는 제거한다.** 수동 테스트는 시리얼 모니터에서
`FIRE 0`을 입력하면 되므로 두 번째 경로를 유지할 이유가 없고, 남겨두면 인덱싱 규칙이
둘이 된다.

### 7.7 펌웨어 변경 범위

`angleToPulse`, `moveServo`, `SERVO_CH`, `HOME_ANGLE`, `MOVE_ANGLE`, `moved[]`는
변경하지 않는다. 변경되는 것은 `loop()`와 `setup()`의 출력부뿐이다.

- 줄 버퍼 누적 (`\n`까지 `char buf[64]`에 축적) — 약 15줄
- `FIRE` / `RESET` / `PING` 분기와 CSV 파싱 — 약 20줄
- `setup()` 끝에 `READY 4` 출력, 기존 안내문에 `#` 접두사
- `Serial.begin(115200)`

### 7.8 인터페이스


## 10. 테스트

### 10.1 단위 테스트 — 하드웨어·카메라·인형 불필요

**`tiles.py`**

- `select_tiles`: 8방향 중심각, 각 구간 경계값 양쪽(예: 22.4° / 22.6°),
  **0°/360° 되감김**, 게이트 조합 3종(R 미달 / lean 미달 / 둘 다).
- 규칙 3(1장): 대각 중심 ±10° 경계 양쪽(34.9° / 35.1°), `τ_R_strict` 경계 양쪽,
  **정방향에서는 `R = 1.0`이어도 절대 1장이 나오지 않는 것**,
  **게이트가 규칙 3을 이긴다는 것**(45° 그리고 `R = 0.99` 그리고 `lean_ratio < τ_lean` → 4장).
- **불변식:** 반환 집합의 크기는 항상 1, 2, 3, 4 중 하나. 빈 집합은 반환되지 않는다.
- `resolve_direction`: **350°와 10°의 평균이 0° 부근으로 나오는지.** 산술평균이면
  180°가 나오며, 이것이 이 함수가 존재하는 이유다. `R`은 동일 각도 N개 → 1.0,
  정반대 두 각 → 0.
- `lean_from_landmarks`: 합성 랜드마크. 직립 → `lean_ratio ≈ 0`,
  우측 기울임 → `direction ≈ 90°`, `camera_yaw_deg` 적용 확인.

**`tile_protocol.py`** — 가짜 시리얼 객체를 주입한다.

- `fire({1,3})`이 정확히 `b"FIRE 1,3\n"`을 전송하는지
- `ERR` 수신 시 예외 없이 `False` 반환
- `READY` 미수신 시 시뮬레이션 모드로 강등
- 타임아웃 시 블로킹이 200ms를 초과하지 않는지

### 10.2 하드웨어 검증 — 시리얼 모니터

아두이노만 필요하다. 카메라, 인형, 파이썬 모두 불필요하다.

```
FIRE 0,1,2,3     ← 가정 A2 검증. 4개 동시 기동에 전원이 버티는지
FIRE 1,3
RESET
PING
FIRE 9           ← ERR 반환 확인
FIRE             ← 인자 없을 때 크래시하지 않는지
```

**`FIRE 0,1,2,3`은 가능한 한 조기에 수행한다.** 어댑터 용량이 부족하면 보드가 리셋되며,
이는 사전에 알면 어댑터를 교체하면 되지만 시연 당일에 알면 복구할 수 없다.

**함께 측정할 것: 서보가 `HOME_ANGLE`(0°) → `MOVE_ANGLE`(60°)까지 이동하는 시간.**
슬로모 촬영으로 측정한다. 인형은 약 0.3초 만에 쓰러지므로, 서보 이동 시간이 이에
근접하면 "낙상 직전 예측"이 시각적으로 성립하지 않는다. 이 값에 따라 시연 서사를
조정해야 한다.

### 10.3 임계값 튜닝 절차

1. `--no-serial`로 8방향 + 수직 주저앉기를 각 3~4회, 총 25~30회 기록한다.
2. 로그의 `lean_ratio` 분포를 확인한다. **방향성 낙상**의 하위 10퍼센타일 **아래**로
   `τ_lean`을 설정한다. 방향성 낙상이 4장으로 강등되지 않아야 한다.
3. 그 값에서 **수직 주저앉기가 실제로 임계값 아래에 떨어지는지** 확인한다.
4. `R`도 동일한 방식으로 설정한다. `τ_R_strict`는 같은 `R` 분포에서 **상위 4분의 1
   정도가 넘어서는 지점**으로 잡는다 — 1장 작동이 드물게, 방향이 특히 깨끗한 회차에만
   나오도록 하는 것이 의도다.
5. **두 분포가 겹치면** — 수직 붕괴와 방향성 낙상의 `lean_ratio`가 분리되지 않으면 —
   게이트로는 구분이 불가능하다는 결론이다. §6.6의 차선책으로 전환한다.

5번이 나올 가능성을 열어둔다. 이 설계가 틀릴 수 있는 지점이고, 데이터로 판별되며,
대안이 준비되어 있다.

## 11. 시연 대비

### 11.1 4단 사다리

| 단 | 방법 | 준비 비용 | 의존 |
|---|---|---|---|
| 1 | **인형 + 카메라** (주 시연) | — | 가정 A1 |
| 2 | **사람 기울이기** (lean-and-recover) | 0 | `--profile human`만 변경 |
| 3 | **녹화 영상 재생** | 리허설 성공 회차 녹화 | 없음 |
| 4 | **시리얼 모니터 수동** `FIRE 1,3` | 0 | 아두이노만 |

**2단이 코드 변경 없이 가능한 것은 §8의 프로파일 분리 덕분이다.** 인형이 인식되지
않으면 즉석에서 사람이 기울이는 방식으로 전환한다.

2단은 안전하며, 오히려 주장을 더 잘 증명한다. 이 시스템은 낙상 **직전** 감지이므로
시연자가 완전히 넘어질 필요가 없다. 몸이 아직 서 있는 상태에서 서보가 작동하는 것을
보이면 "감지가 아니라 예측"이라는 주장이 시각적으로 성립한다.

**3단이 핵심 보험이다.** `python main.py <영상경로>`는 이미 동작한다(`main.py:200`).
리허설이 성공한 회차를 촬영해두는 것만으로 비용 0의 보험이 생긴다.

### 11.2 시연 전 점검 목록

- 시리얼 포트를 `--port`로 명시하고 `READY` 응답을 확인
- `calibration.json` 존재, `rows × cols`가 `READY n`과 일치
- 프로파일 값이 리허설에서 확정한 값과 동일
- 3단 백업 영상 파일 존재 확인
- 4개 동시 기동 재확인(전원 환경이 시연장에서 바뀔 수 있음)
